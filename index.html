Using ParrelSync to Simulate Multiple Players & FPS Setup

Project Blog DH2650 - Setting up the Multiplayer Foundation

1. Decide Game Topic & Technical Stack

For our project, we are building a multiplayer experience where players interact on a moving ship affected by ocean waves. To handle the networking architecture, we decided to use Unity’s Netcode for GameObjects (NGO). For local testing, we are utilizing ParrelSync, which is an absolute lifesaver. It allows us to open a clone of the project editor to test Host/Client interactions seamlessly without needing to build the game every time.

2. Establishing the NetworkManager & Syncing the Environment

The core of the NGO setup is the NetworkManager combined with Unity Transport. Our first technical milestone was getting the ocean physics to sync.
By registering our Ship as a Network Prefab and attaching both a NetworkObject and NetworkTransform, we ensured that the wave-driven physics calculated on the Host are perfectly replicated across all Client instances. It’s a highly satisfying visual validation of state consistency in our distributed environment.

3. First-Person Player Setup & “Owner-Driven” Cameras

Spawning players into a multiplayer scene introduces a classic problem: if every player prefab has an active camera and AudioListener, the client screens will conflict and glitch out.
To solve this, we implemented an ownership-check script (PlayerSetup.cs). By utilizing OnNetworkSpawn() and checking IsOwner, we ensured strict isolation:

  • Local Players: Activate their own camera, lock the cursor, and handle their own input.
  • Remote Players: Keep their cameras and listeners disabled to prevent them from hijacking the local screen.

4. Overcoming the “Server Authority” Trap

When implementing the WASD and mouse-look movement (PlayerMove.cs with a CharacterController), we hit a wall: the Host could move perfectly, but the Client was stuck twitching in place.
This was a harsh but necessary lesson in Server Authority. The NetworkTransform component defaults to Server authority. As a result, the Client’s local inputs were constantly being overridden and rolled back by the Server’s absolute state. By switching the Authority Mode to Owner, we successfully decentralized the movement control, granting clients the authority to drive their own positional state while broadcasting it to the network.

5. Optimizing the ParrelSync Dual-Window Workflow

Testing a First-Person game with locked cursors in two ParrelSync windows can lead to input chaos (e.g., moving the mouse in one window rotates the camera in the other).
We optimized our debugging workflow by implementing a window focus check:
if (!Application.isFocused || Cursor.lockState != CursorLockMode.Locked) return;
Combined with an Escape-key toggle to unlock the mouse, the scripts now simply ignore input if the specific game window loses focus. This makes dual-window testing perfectly smooth and isolated.

Next Steps: With the multiplayer foundation and FPS controllers fully synced, the next challenge is physics-based: ensuring the players can stand firmly on the pitching and rolling deck of the ship without sliding into the ocean.
