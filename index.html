AddingPauseMenu

Today we had a team sync to align our milestones and distribute upcoming tasks. My primary objective for this sprint was to implement a functional Pause Menu that doubles as a network diagnostic tool for our multi-platform testing.

I successfully built the UI overlay featuring a semi-transparent gray filter and real-time network statistics, including Local IP, Host IP, and a Ping indicator. To ensure the menu doesn’t interfere with our initial connection screen, I implemented a logic gate using NetworkManager.Singleton.IsListening. This ensures the Tab key only activates the menu once a network session is actually live, keeping the starting interface clean.

The main technical hurdle was resolving the “input war” between the UI and my existing PlayerMove.cs. I handled this by utilizing a static IsPaused flag to act as a master switch. Now, when the menu is active, it completely intercepts player movement and rotation logic. This fixed a critical bug where mouse clicks were being “stolen” by the player controller, which previously prevented buttons from responding. The system is now robust, allowing for seamless transitions between gameplay and menu navigation. All changes have been pushed to the feature branch for further integration.

CreateGithubPage & Core Multiplayer Architecture

Devlog: Laying the Foundation for LAN Multiplayer Collaboration

Today’s core objective was to transition our project from a standalone single-player demo into a robust multiplayer collaborative engineering effort. We refactored all core scripts and established a standardized distributed version control workflow.

1. Core Network Architecture (NGO Implementation)

We adopted Unity’s Netcode for GameObjects (NGO) framework and deeply customized it for the flexibility of Local Area Network (LAN) multiplayer:

  • Dynamic IP Connection: Refactored NetworkUI.cs. By introducing the UnityTransport component, we enabled dynamic modification of ConnectionData at runtime. Clients can now directly input the Host’s LAN IP via the UI to connect, completely eliminating the pain point of hardcoding localhost for LAN testing.
  • Client-Side Component Isolation: Wrote PlayerSetup.cs. Using the OnNetworkSpawn hook, we dynamically toggle the Camera and AudioListener based on the IsOwner status. This ensures that in a multiplayer environment, each player only sees the world through their own perspective, avoiding Unity rendering and audio conflicts.

2. Physics & Visuals Synergy (Buoyancy & Waves)

To handle ocean wave synchronization, we opted for a low-bandwidth, high-performance deterministic algorithm:

  • Deterministic Waves: WaveManager.cs uses the global absolute time Time.time to drive the sine wave formula. Because this mathematical formula is identical across all connected Clients, we achieve perfect visual alignment purely through local calculations, without needing to sync thousands of vertex coordinates over the network.
  • Server-Authoritative Buoyancy: Floater.cs strictly adheres to server-authoritative logic. Physics forces (AddForceAtPosition) are calculated ONLY on the Server/Host. The resulting transform data is then synced across the network via NetworkTransform, ensuring the ship’s bobbing motion is consistent and stutter-free for all players.

3. Environment & Spawn Logic

To improve the testing experience, we added a “Coast” module to the scene:

  • Static Anchor Point: Created a PlayerSpawnPoint and adjusted the default spawn coordinates of the player prefab. This allows joining players to safely land on solid ground to observe the waves, rather than spawning directly into the deep ocean.

4. Engineering & Version Control (Git / GitHub)

To kick off team collaboration, we officially set up the remote repository with “industry-standard” safeguards today:

  • Git LFS Configuration: Pre-configured Git Large File Storage to track large assets like *.fbx and *.png. This prevents our .git repository from bloating when we introduce high-poly ship models later.
  • Standardized .gitignore: Created a Unity-specific .gitignore (with English comments) to strictly filter out local cache directories like Library and Temp.
  • Code Standardization: Upgraded all core scripts (Floater, NetworkUI, PlayerMove, PlayerSetup, WaterMeshGenerator, WaveManager) with professional English comments and added File Headers to clarify each script’s role within our distributed architecture.

5. Next Steps (Roadmap)

  • Optimize player physics/attachment logic on the ship (addressing relative transform synchronization).
  • Introduce the first-pass 3D ship model to replace the current placeholder Cubes.
  • Conduct the first cross-platform (Mac/Windows) multiplayer stress test once the team pulls the code.

Current Status: 🏗️ Base architecture pushed to GitHub. Ready for team collaboration.

Using ParrelSync to Simulate Multiple Players & FPS Setup

Project Blog DH2650 - Setting up the Multiplayer Foundation

1. Decide Game Topic & Technical Stack

For our project, we are building a multiplayer experience where players interact on a moving ship affected by ocean waves. To handle the networking architecture, we decided to use Unity’s Netcode for GameObjects (NGO). For local testing, we are utilizing ParrelSync, which is an absolute lifesaver. It allows us to open a clone of the project editor to test Host/Client interactions seamlessly without needing to build the game every time.

2. Establishing the NetworkManager & Syncing the Environment

The core of the NGO setup is the NetworkManager combined with Unity Transport. Our first technical milestone was getting the ocean physics to sync.
By registering our Ship as a Network Prefab and attaching both a NetworkObject and NetworkTransform, we ensured that the wave-driven physics calculated on the Host are perfectly replicated across all Client instances. It’s a highly satisfying visual validation of state consistency in our distributed environment.

3. First-Person Player Setup & “Owner-Driven” Cameras

Spawning players into a multiplayer scene introduces a classic problem: if every player prefab has an active camera and AudioListener, the client screens will conflict and glitch out.
To solve this, we implemented an ownership-check script (PlayerSetup.cs). By utilizing OnNetworkSpawn() and checking IsOwner, we ensured strict isolation:

  • Local Players: Activate their own camera, lock the cursor, and handle their own input.
  • Remote Players: Keep their cameras and listeners disabled to prevent them from hijacking the local screen.

4. Overcoming the “Server Authority” Trap

When implementing the WASD and mouse-look movement (PlayerMove.cs with a CharacterController), we hit a wall: the Host could move perfectly, but the Client was stuck twitching in place.
This was a harsh but necessary lesson in Server Authority. The NetworkTransform component defaults to Server authority. As a result, the Client’s local inputs were constantly being overridden and rolled back by the Server’s absolute state. By switching the Authority Mode to Owner, we successfully decentralized the movement control, granting clients the authority to drive their own positional state while broadcasting it to the network.

5. Optimizing the ParrelSync Dual-Window Workflow

Testing a First-Person game with locked cursors in two ParrelSync windows can lead to input chaos (e.g., moving the mouse in one window rotates the camera in the other).
We optimized our debugging workflow by implementing a window focus check:
if (!Application.isFocused || Cursor.lockState != CursorLockMode.Locked) return;
Combined with an Escape-key toggle to unlock the mouse, the scripts now simply ignore input if the specific game window loses focus. This makes dual-window testing perfectly smooth and isolated.

Next Steps: With the multiplayer foundation and FPS controllers fully synced, the next challenge is physics-based: ensuring the players can stand firmly on the pitching and rolling deck of the ship without sliding into the ocean.
