CreateGithubPage & Core Multiplayer Architecture

Devlog: Laying the Foundation for LAN Multiplayer Collaboration

Today’s core objective was to transition our project from a standalone single-player demo into a robust multiplayer collaborative engineering effort. We refactored all core scripts and established a standardized distributed version control workflow.

1. Core Network Architecture (NGO Implementation)

We adopted Unity’s Netcode for GameObjects (NGO) framework and deeply customized it for the flexibility of Local Area Network (LAN) multiplayer:

  • Dynamic IP Connection: Refactored NetworkUI.cs. By introducing the UnityTransport component, we enabled dynamic modification of ConnectionData at runtime. Clients can now directly input the Host’s LAN IP via the UI to connect, completely eliminating the pain point of hardcoding localhost for LAN testing.
  • Client-Side Component Isolation: Wrote PlayerSetup.cs. Using the OnNetworkSpawn hook, we dynamically toggle the Camera and AudioListener based on the IsOwner status. This ensures that in a multiplayer environment, each player only sees the world through their own perspective, avoiding Unity rendering and audio conflicts.

2. Physics & Visuals Synergy (Buoyancy & Waves)

To handle ocean wave synchronization, we opted for a low-bandwidth, high-performance deterministic algorithm:

  • Deterministic Waves: WaveManager.cs uses the global absolute time Time.time to drive the sine wave formula. Because this mathematical formula is identical across all connected Clients, we achieve perfect visual alignment purely through local calculations, without needing to sync thousands of vertex coordinates over the network.
  • Server-Authoritative Buoyancy: Floater.cs strictly adheres to server-authoritative logic. Physics forces (AddForceAtPosition) are calculated ONLY on the Server/Host. The resulting transform data is then synced across the network via NetworkTransform, ensuring the ship’s bobbing motion is consistent and stutter-free for all players.

3. Environment & Spawn Logic

To improve the testing experience, we added a “Coast” module to the scene:

  • Static Anchor Point: Created a PlayerSpawnPoint and adjusted the default spawn coordinates of the player prefab. This allows joining players to safely land on solid ground to observe the waves, rather than spawning directly into the deep ocean.

4. Engineering & Version Control (Git / GitHub)

To kick off team collaboration, we officially set up the remote repository with “industry-standard” safeguards today:

  • Git LFS Configuration: Pre-configured Git Large File Storage to track large assets like *.fbx and *.png. This prevents our .git repository from bloating when we introduce high-poly ship models later.
  • Standardized .gitignore: Created a Unity-specific .gitignore (with English comments) to strictly filter out local cache directories like Library and Temp.
  • Code Standardization: Upgraded all core scripts (Floater, NetworkUI, PlayerMove, PlayerSetup, WaterMeshGenerator, WaveManager) with professional English comments and added File Headers to clarify each script’s role within our distributed architecture.

5. Next Steps (Roadmap)

  • Optimize player physics/attachment logic on the ship (addressing relative transform synchronization).
  • Introduce the first-pass 3D ship model to replace the current placeholder Cubes.
  • Conduct the first cross-platform (Mac/Windows) multiplayer stress test once the team pulls the code.

Current Status: 🏗️ Base architecture pushed to GitHub. Ready for team collaboration.
